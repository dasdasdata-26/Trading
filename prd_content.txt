1. Product Overview
A locally-hosted web application that automates generation of professional client proposals for HPM Systems, Inc. The user fills out a structured form in their web browser, then either downloads the resulting .docx to open in Microsoft Word for review and manual editing, or exports directly to PDF from within the app when no edits are needed. The app runs on the user's own computer — no cloud dependency, no subscription, no registry changes required.
To start the app, the user runs: python app.py from the app directory. The server starts on localhost:5000 and the app opens automatically in the default browser.
⚠ MULTI-TEMPLATE ARCHITECTURE — Critical constraint: This PRD covers Template 1 (Gas Detection / Controls). The app must be data-driven so that adding Template 2, 3, etc. in future requires only a new JSON config file — zero code changes. See Section 7.
Each generated proposal includes:
HPM Systems logo, top-left, page 1 only
Plain-text continuation header on pages 2+ (7pt Arial)
Addressee and project identification block
Four editable Roman-numeral scope sections with multi-level sub-items (start empty each proposal)
Assumptions/Exclusions section with lettered items (pre-loaded from template defaults)
Professional Fee table with dot leaders and yellow-highlighted TOTAL row (whole dollars, no cents)
Closing paragraph and two-column signature block with toggleable Cc line
Company Rate Schedule as a separate final page (auto-populated from Settings)
Centered 7pt footer on every page

5. Functional Requirements
5.1 Application Shell
Single-page web application served from Flask on localhost:5000. Scrollable form with collapsible section panels. Minimum usable width: 900px. Navigation/menu bar: New Proposal, Save Draft, Load Draft, Settings, About. Persistent action toolbar: Download .docx, Export to PDF, Save Draft, Load Draft, New Proposal, Settings.
'New Proposal' shows a confirmation dialog if there is unsaved work.
5.2 Template Selection
On 'New Proposal', the user selects a proposal template from a dropdown. In v1.0 only Template 1 exists, but the selector must be present and functional so adding Template 2 later requires no UI changes. The selected template loads its default section headings, default assumptions, default intro, and default closing paragraph.
5.3 Data Entry Form — Proposal Identification & Addressee
Required fields (*) are validated before document generation. Inline field-level error messages shown on attempted generation.
Proposal Identification:
Date * — date picker, defaults to today, output format: 'Month DD, YYYY' (e.g., February 22, 2026)
Proposal Number * — free-text field, validated against: six digits, hyphen, literal 'rev', one or more digits (regex: ^\d{6}-rev\d+$). Example: 226001-rev0. User always enters manually. No auto-increment.
Addressee Block:
Recipient First Name * — used in salutation 'Dear [First Name],'
Recipient Last Name * — used in full addressee block
Company Name * — also appears in continuation header on pages 2+
Street Address *
City, State, ZIP *
Subject Line * — short project description; appears in RE: block and continuation header
Site / Location * — physical site; appears in RE: block and continuation header
5.4 Scope of Work — Four Section Panels
Four Roman-numeral sections, each in its own collapsible panel. On every new proposal, all four sections start with an empty sub-item list — no pre-loaded defaults. The user builds the scope from scratch each time.
Section headings are editable per proposal (current proposal only; does not update template defaults). A 'Reset to Default' button next to each heading restores the template default.
Template 1 — Default Section Headings:
I — DESIGN, PROJECT MANAGEMENT & CONTRACTOR COORDINATION
II — EQUIPMENT
III — PLC & HMI PROGRAMMING, START-UP, TESTING & INSPECTIONS
IV — ELECTRICAL INSTALLATION
Sub-item management (per section):
Add new sub-item (button appends to bottom of list)
Delete sub-item (with confirmation if the item has content)
Reorder sub-items via drag-and-drop; Up/Down keyboard shortcuts as fallback
Each sub-item supports nested numbered sub-sub-items (1, 2, 3...) with the same add/delete/reorder controls
Sub-item body text editor supports: bold, italic, underline, and inline bullet sub-lists
Output formatting in .docx:
Section header: 9pt Arial, bold — 'I) [SECTION HEADING]'
Lettered sub-items: 9pt Arial, indented ~0.5", auto-labeled — 'A) [text]'
Numbered sub-sub-items: 9pt Arial, indented ~1.0", auto-labeled — '1. [text]'
5.5 Assumptions & Exclusions
Fixed section heading: 'ASSUMPTIONS/EXCLUSIONS' (bold, underlined in output). Pre-loaded from the active template's default assumption list stored in Settings. The user may add, delete, or reorder items per proposal via drag-and-drop. Items are auto-labeled A, B, C... in the output.
Pre-loaded intro line (not editable in the form, fixed): 'The scope of work is based on the following assumptions and/or exclusions:'
Template 1 default assumptions list: see Section 9.1. Owner to confirm final list before development begins.
5.6 Professional Fee Summary
Four currency input fields (one per section) plus a read-only TOTAL FEE field that auto-sums on any change. Whole integers only — no cents. Field accepts plain integer input; formats on blur with $ prefix and comma separators (e.g., user types 12960 → field displays $12,960). TOTAL displays in the same format.
⚠ Future version: add an 'Import Fees from Excel' button that reads four totals from a user-selected .xlsx and populates the fields. The fee data model must not prevent this addition.
Output .docx fee table:
Each row: Section number + heading, dot leaders across full width, right-aligned whole-dollar amount
TOTAL FEE row: bold, yellow highlight
Dollar signs appear on every row including individual section rows (e.g., $12,960) and the TOTAL FEE row (e.g., $158,320) — CONFIRMED
Example row: 'I) DESIGN, PROJECT MANAGEMENT & CONTRACTOR COORDINATION ............... $12,960'
5.7 Closing Paragraph & Signature Block
Closing paragraph: pre-loaded from template Settings, editable per proposal in a text area.
Signature block (two-column layout, matching sample document):
Left column: 'Sincerely,' / Blank signature line / [User Name from Settings] / [User Title from Settings] / 'HPM Systems, Inc.' / 'CA contractor license # [number from Settings]' / Cc line (see below)
Right column: 'Client' / 'Authorized Representative' / Blank signature line + 'Date' label / 'P.O.# (if applicable)'
Cc line behavior:
Toggle checkbox in the form: 'Include Cc line' (defaults to checked)
When checked, a text field shows the Cc line content, pre-filled from Settings default (e.g., 'Cc: Bert Buehler, President'), editable per proposal
When unchecked, the Cc line is omitted from the output entirely
5.8 Rate Schedule (Separate Final Page)
Begins with a mandatory page break. Content is auto-populated from the rate schedule stored in Settings. The user does not need to do anything per-proposal — the rate schedule is always appended automatically.
Annual update workflow: User opens Settings, edits the relevant fields (year, individual rates, special charges) manually, or pastes updated text into the bulk text area. Both editing modes stay in sync. See Section 9.2 for the default content.
Rate schedule sections in output:

7. Multi-Template Architecture (Critical Design Constraint)
⚠ Claude Code must read and implement this section. The template system is not optional — it is a core architectural requirement.
HPM Systems uses multiple proposal formats for different service lines. All share the same structure (four Roman-numeral sections, assumptions, fee table, signature block, rate schedule page) but differ in default headings and canned text. Templates are defined in JSON config files, not in Python code.
Rule: Adding Template 2 in a future version must require only creating a new JSON config file and adding its filename to a template registry. Zero code changes to the application.
Template JSON schema (minimum required fields):
template_id — string, unique key (e.g., "gas_detection_v1")
template_name — display name (e.g., "Gas Detection / Controls")
section_headings — array of exactly 4 strings (default headings for I–IV)
default_intro — string (intro paragraph)
default_assumptions — ordered array of strings (one per lettered item)
default_closing — string (closing paragraph)
Note: default scope sub-items are intentionally absent from the schema — all sections start empty on every new proposal.
Template registry: a simple JSON array file (templates.json) listing the filenames of all available template configs. The app reads this on startup to populate the template selector.
