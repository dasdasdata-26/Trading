1. PRODUCT OVERVIEW
================================================================================

A locally-hosted web application that automates generation of professional
client proposals for HPM Systems, Inc. The user fills out a structured form in
their web browser, then either downloads the resulting .docx to open in
Microsoft Word for review and manual editing, or exports directly to PDF from
within the app when no edits are needed. The app runs on the user's own
computer — no cloud dependency, no subscription, no registry changes required.

To start the app, the user runs:
    python app.py
from the app directory. The server starts on localhost:5000 and the app opens
automatically in the default browser. Alternatively, double-click run.bat on
Windows.

⚠ MULTI-TEMPLATE ARCHITECTURE — Critical constraint: The app is data-driven
so that adding Template 2, 3, etc. requires only a new JSON config file —
zero code changes. See Section 7. Multiple additional templates are actively
planned for v2.x.

Each generated proposal includes:
  • HPM Systems logo, top-left, page 1 only
  • Plain-text continuation header on pages 2+ (7pt Arial)
  • Addressee and project identification block
  • Up to four editable Roman-numeral scope sections with multi-level sub-items
    (sections can be individually deleted per proposal)
  • Assumptions/Exclusions section with lettered items (pre-loaded from
    template defaults)
  • Professional Fee table with dot leaders and yellow-highlighted TOTAL row
    (whole dollars, no cents)
  • Closing paragraph and two-column signature block with toggleable Cc line
  • Company Rate Schedule as a separate final page (auto-populated from
    Settings)
  • Centered 7pt footer on every page

5. FUNCTIONAL REQUIREMENTS
================================================================================

5.1 Application Shell
---------------------
Single-page web application served from Flask on localhost:5000. Scrollable
form with collapsible section panels. Minimum usable width: 900px.

Navigation bar: New Proposal, Save Draft, Load Draft, Settings, About.
Persistent action toolbar: Download .docx, Export to PDF, Save Draft,
  Load Draft, New Proposal, Settings.

'New Proposal' shows a confirmation dialog if there is unsaved work.


5.2 Template Selection
-----------------------
On 'New Proposal', the user selects a proposal template from a dropdown.
The selected template loads its default section headings, default assumptions,
default intro, and default closing paragraph. Adding a new template requires
only creating a JSON config file and registering it — see Section 7.


5.3 Data Entry Form — Proposal Identification & Addressee
----------------------------------------------------------
Required fields (*) are validated before document generation. Inline
field-level error messages shown on attempted generation.

Proposal Identification:
  Date * — date picker, defaults to today
            output format: 'Month DD, YYYY' (e.g., February 22, 2026)
  Proposal Number * — validated: ^\d{6}-rev\d+$
                      Example: 226001-rev0. Manual entry always.

Addressee Block:
  Recipient First Name * — used in salutation 'Dear [First Name],'
  Recipient Last Name *  — used in full addressee block
  Company Name *         — also in continuation header pages 2+
  Street Address *
  City, State, ZIP *
  Subject Line *         — in RE: block and continuation header
  Site / Location *      — in RE: block and continuation header


5.4 Scope of Work — Section Panels (I–IV)
------------------------------------------
Up to four Roman-numeral sections, each in its own collapsible panel. On every
new proposal, all sections start with an empty sub-item list.

Section headings are editable per proposal. A 'Reset to Default' button
restores the template default heading.

DELETE / RESTORE (added v2.1):
  Each section header row has a '✕ Delete' button. Clicking it:
    — Hides the section panel from the UI (section-deleted CSS class)
    — Excludes the section entirely from the .docx output
    — Shows a 'Restore' button in a bar below the sections list
  Clicking 'Restore [numeral])' reverses the deletion.
  Delete/restore state is preserved in Save Draft and restored on Load Draft.
  'New Proposal' resets all sections to visible.
  Only non-deleted sections appear in the PROFESSIONAL FEE table; only their
  fees are summed in the TOTAL.

Sub-item management (per section):
  • Add new sub-item (button appends to bottom of list)
  • Delete sub-item (with confirmation if the item has content)
  • Reorder via drag-and-drop; Up/Down arrow buttons as fallback
  • Each sub-item supports nested numbered sub-sub-items with the same
    add/delete/reorder controls
  • Sub-item body text editor: bold, italic, underline, inline bullet sub-lists

Output formatting in .docx (exact measurements — see also Section 8.2):
  Section header:    9pt Arial, bold — 'I) [SECTION HEADING]'
                     Roman numeral at left margin (0"), heading text at 0.5"
  Lettered sub-items:   9pt Arial — 'A) [text]'
                        Label at 0.5", text wraps at 1.0"
  Numbered sub-sub-items: 9pt Arial — '1) [text]'   ← format is "1)" not "1."
                          Label at 1.0", text wraps at 1.5"

Word auto-numbering (added v2.1):
  The .docx uses OOXML abstractNum/num list definitions so that when the user
  opens the file in Word and adds new items after the exported content, Word
  automatically continues the letter/number sequence (E, F, G... or 4, 5, 6).
  Each section's letter list uses a separate concrete numId so that section
  II starts at A regardless of how many items section I has.


5.5 Assumptions & Exclusions
-----------------------------
Fixed heading: 'ASSUMPTIONS/EXCLUSIONS' (bold, underlined). Pre-loaded from
the active template's default assumption list. User may add, delete, or
reorder per proposal. Items are auto-labeled A, B, C... in the output.

Fixed intro line: 'The scope of work is based on the following assumptions
and/or exclusions:'


5.6 Professional Fee Summary
-----------------------------
Four currency input fields (one per section) plus a read-only TOTAL FEE field
that auto-sums. Whole integers only — no cents.

Input: user types 12960 → formats on blur as $12,960.

⚠ Only non-deleted sections appear in the fee table and contribute to TOTAL.

Output .docx fee table:
  Each row: Section number + heading, dot leaders, right-aligned dollar amount
  TOTAL FEE row: bold, yellow highlight
  Dollar signs on every row including individual section rows.
  Example: 'I) DESIGN, PROJECT MANAGEMENT... ............ $12,960'

⚠ Future version: 'Import Fees from Excel' button (deferred — architecture
must not prevent this addition).


5.7 Closing Paragraph & Signature Block
----------------------------------------
Closing paragraph: pre-loaded from template Settings, editable per proposal.

Signature block (two-column, borderless table):
  Left column:  'Sincerely,' / Signature image or blank space / [User Name] /
                [User Title] / 'HPM Systems, Inc.' /
                'CA contractor license # [number]' / Cc line (see below)
  Right column: 'Client' / 'Authorized Representative' / blank / blank /
                'Signature                Date' / blank / 'P.O.# (if applicable)'

Cc line behavior:
  Toggle checkbox defaults to checked; text pre-filled from Settings default.
  When unchecked, Cc line is omitted entirely from output.


5.8 Rate Schedule (Separate Final Page)
-----------------------------------------
Begins with a mandatory page break. Auto-populated from Settings. No per-
proposal action needed.

Rate schedule sections in output:
  'HPM SYSTEMS RATE SCHEDULE'  — bold heading
  Subtitle: 'The following fee schedule applies through December 31, [year]'
  PROFESSIONAL FEES — role names and hourly rates (body paragraph per role)
  SPECIAL CHARGES  — round bullet (•) list  ← v2.1: bullets, not letters
  Calibration Gas and Service Van per-use charges
  REIMBURSABLE EXPENSES — mileage and reproduction rates


5.9 Page Layout, Header & Footer
----------------------------------
All text: Arial. Body: 9pt. Continuation header and footer: 7pt.

Page 1 Header:
  HPM Systems logo (PNG), top-left, approximately 1.5" wide.

Pages 2+ Continuation Header (7pt Arial, left-aligned):
  [Client Company Name]
  Subject: [Subject Line]
  Site: [Site]
  Proposal [Proposal Number]
  [Date]
  Page X of Y

Footer (all pages, 7pt Arial, centered):
  '1292 Kifer Road, Suite 807, Sunnyvale, CA 94086 ♦ Ph: (408) 615-6900'
  Configurable in Settings.

Page Setup: US Letter (8.5" × 11"), 1" margins all sides.

7. MULTI-TEMPLATE ARCHITECTURE
================================================================================

⚠ Core architectural requirement — not optional.

HPM Systems uses multiple proposal formats for different service lines. All
share the same structure (four Roman-numeral sections, assumptions, fee table,
signature block, rate schedule) but differ in default headings and canned text.

Rule: Adding a new template requires ONLY:
  1. Creating a new JSON config file in hpm_proposal/templates/
  2. Adding its filename to hpm_proposal/templates/templates.json
  Zero code changes to app.py, document_generator.py, or any HTML/JS file.

--------------------------------------------------------------------------------
7.1 Template JSON Schema (required fields)
--------------------------------------------------------------------------------

{
  "template_id":          "unique_string_key",
  "template_name":        "Display Name shown in UI dropdown",
  "section_headings":     [
                            "Default heading for section I",
                            "Default heading for section II",
                            "Default heading for section III",
                            "Default heading for section IV"
                          ],
  "default_intro":        "Opening paragraph text shown after salutation.",
  "default_closing":      "Closing paragraph. Use {user_email} as a placeholder
                           and it will be substituted from Settings at load time.",
  "default_assumptions":  [
                            "First assumption item text.",
                            "Second assumption item text.",
                            "..."
                          ]
}

Notes:
  • template_id must be unique across all templates (e.g., "fire_alarm_v1")
  • section_headings must be an array of exactly 4 strings
  • default_assumptions is an ordered array; items labeled A, B, C... in output
  • default scope sub-items are intentionally absent — sections always start
    empty on every new proposal
  • {user_email} in default_closing is the only supported placeholder

--------------------------------------------------------------------------------
7.2 Template Registry File
--------------------------------------------------------------------------------

File: hpm_proposal/templates/templates.json
Format: a simple JSON array of filenames.

Example:
[
  "gas_detection_v1.json",
  "fire_alarm_v1.json",
  "commissioning_v1.json"
]

The app reads this file on startup to populate the template selector dropdown.
Templates appear in the dropdown in the order they are listed.

--------------------------------------------------------------------------------
7.3 Step-by-Step: Adding a New Template
--------------------------------------------------------------------------------

Step 1.  Copy gas_detection_v1.json and rename it, e.g., fire_alarm_v1.json

Step 2.  Edit the new file:
           - Set "template_id" to a new unique key (e.g., "fire_alarm_v1")
           - Set "template_name" to the display name for the dropdown
           - Replace all four "section_headings" with the headings for the
             new service line
           - Replace "default_intro" with the appropriate opening paragraph
           - Replace "default_assumptions" with the appropriate list items
           - Replace "default_closing" with the appropriate closing paragraph

Step 3.  Open hpm_proposal/templates/templates.json and add the new filename
         to the array.

Step 4.  Restart the app (python app.py). The new template will appear in the
         dropdown.

Step 5.  Test by selecting the new template, verifying headings load, and
         downloading a sample .docx.

Step 6.  Run: git add . && git commit -m "Add [template name] template"
         to save the new template in version control.

8.2 Typography & Formatting Reference
---------------------------------------

Element                      | Style
-----------------------------|--------------------------------------------------
All body text                | Arial 9pt
Continuation header (pp 2+)  | Arial 7pt, left-aligned
Footer text                  | Arial 7pt, centered
Date on page 1               | Arial 9pt, bold
'SCOPE OF WORK' heading      | Arial 9pt, bold, underlined
Section headers I–IV         | Arial 9pt, bold
Lettered sub-items (A, B...) | Arial 9pt
Numbered sub-sub-items (1))  | Arial 9pt
'ASSUMPTIONS/EXCLUSIONS'     | Arial 9pt, bold, underlined
'PROFESSIONAL FEE' heading   | Arial 9pt, bold, underlined
Fee table rows               | Arial 9pt, dot leaders, right-aligned $
TOTAL FEE row                | Arial 9pt, bold, yellow background shading
Rate Schedule headings       | Arial 9–10pt, bold

LINE SPACING (fixed in v2.1):
  All paragraphs: single line spacing (w:line=240, w:lineRule=auto)
  Space before: 0pt on all paragraphs
  Space after:  0pt on all paragraphs
  Applied to: Normal style definition AND every paragraph element in the
  document via direct OOXML XML to guarantee Word renders correctly.

INDENTATION — Exact Measurements (fixed in v2.1):
  Roman numeral section headers:
    • Label (e.g., "I)") at left margin — 0" from text area left edge
    • Tab stop at 0.5"; heading text starts at 0.5"
    • Implemented as: w:left=720, w:hanging=720 (twips)

  Lettered sub-items (A, B, C...):
    • Label (e.g., "A)") at 0.5" from text area left edge
    • Text wraps at 1.0"
    • Implemented as: w:left=1440, w:hanging=720

  Numbered sub-sub-items (1), 2), 3)...):
    • Label (e.g., "1)") at 1.0" from text area left edge
    • Text wraps at 1.5"
    • Implemented as: w:left=2160, w:hanging=720

WORD AUTO-NUMBERING:
  Letter lists use numFmt="upperLetter", lvlText="%1)"
  Number lists use numFmt="decimal",     lvlText="%1)"
  Each section's letter list gets a separate OOXML concrete numId with
  startOverride=1 so it resets to A independently.
  Numbered sub-lists also reset per parent letter item.
